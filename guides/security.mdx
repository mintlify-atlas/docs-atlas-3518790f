Membrane provides multiple layers of security to protect your agent's memory substrate in production deployments.

## Security Model

Membrane's security architecture includes:

- **Encryption at Rest** -- SQLCipher encryption for database files
- **TLS Transport** -- Encrypted gRPC connections
- **Authentication** -- API key-based client authentication
- **Rate Limiting** -- Token bucket rate limiting per client
- **Trust-Aware Retrieval** -- Sensitivity-based access control
- **Input Validation** -- Payload size limits, string length checks, NaN/Inf rejection

## Encryption at Rest

Membrane uses [SQLCipher](https://www.zetetic.net/sqlcipher/) to encrypt the database file on disk.

### Enabling Encryption

<Warning>
  Store encryption keys in environment variables, never in config files or version control.
</Warning>

<Steps>
  <Step title="Generate a strong encryption key">
    ```bash
    # Generate a 32-character key
    openssl rand -base64 32
    ```
  </Step>

  <Step title="Set the environment variable">
    ```bash
    export MEMBRANE_ENCRYPTION_KEY="your-32-char-encryption-key-here"
    ```
  </Step>

  <Step title="Start the daemon">
    ```bash
    ./bin/membraned --config config.yaml
    ```

    The daemon reads `MEMBRANE_ENCRYPTION_KEY` automatically.
  </Step>
</Steps>

### Go Library

When using Membrane as an embedded library:

```go
import (
    "os"
    "github.com/GustyCube/membrane/pkg/membrane"
)

cfg := membrane.DefaultConfig()
cfg.DBPath = "my-agent.db"
cfg.EncryptionKey = os.Getenv("MEMBRANE_ENCRYPTION_KEY")

m, err := membrane.New(cfg)
if err != nil {
    log.Fatal(err)
}
defer m.Stop()
```

### Key Rotation

To rotate encryption keys:

1. Stop the daemon
2. Decrypt the database with the old key
3. Re-encrypt with the new key using SQLCipher's `sqlcipher_export()`
4. Restart with the new key

## TLS Transport

Enable TLS to encrypt gRPC traffic between clients and the daemon.

### Generating TLS Certificates

<Tabs>
  <Tab title="Self-Signed (Development)">
    ```bash
    # Generate a self-signed certificate
    openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -days 365 -nodes
    ```
  </Tab>

  <Tab title="Let's Encrypt (Production)">
    ```bash
    # Use certbot to obtain a certificate
    sudo certbot certonly --standalone -d membrane.example.com
    ```

    Certificates will be in `/etc/letsencrypt/live/membrane.example.com/`.
  </Tab>
</Tabs>

### Enabling TLS

Add TLS settings to your config:

```yaml
listen_addr: ":9090"
tls_cert_file: "/etc/membrane/tls/cert.pem"
tls_key_file: "/etc/membrane/tls/key.pem"
```

Start the daemon:

```bash
./bin/membraned --config config.yaml
```

### Client Configuration

<CodeGroup>
```ts TypeScript
import { MembraneClient } from "@gustycube/membrane";

const client = new MembraneClient("membrane.example.com:443", {
  tls: true,
  tlsCaCertPath: "/path/to/ca.pem",  // Optional: custom CA
  apiKey: process.env.MEMBRANE_API_KEY,
});
```

```python Python
from membrane import MembraneClient

client = MembraneClient(
    "membrane.example.com:443",
    tls=True,
    tls_ca_cert="/path/to/ca.pem",  # Optional: custom CA
    api_key=os.getenv("MEMBRANE_API_KEY"),
)
```
</CodeGroup>

## Authentication

Membrane supports API key-based authentication using Bearer tokens.

### Enabling Authentication

<Steps>
  <Step title="Generate an API key">
    ```bash
    # Generate a random API key
    openssl rand -hex 32
    ```
  </Step>

  <Step title="Set the environment variable">
    ```bash
    export MEMBRANE_API_KEY="your-secret-api-key"
    ```
  </Step>

  <Step title="Start the daemon">
    ```bash
    ./bin/membraned --config config.yaml
    ```
  </Step>
</Steps>

### Client Authentication

<CodeGroup>
```ts TypeScript
import { MembraneClient } from "@gustycube/membrane";

const client = new MembraneClient("localhost:9090", {
  apiKey: process.env.MEMBRANE_API_KEY,
});
```

```python Python
import os
from membrane import MembraneClient

client = MembraneClient(
    "localhost:9090",
    api_key=os.getenv("MEMBRANE_API_KEY"),
)
```
</CodeGroup>

### Authentication Failures

If the API key is missing or invalid, requests fail with `UNAUTHENTICATED` status:

```
grpc error: code=16 (UNAUTHENTICATED) message="invalid API key"
```

## Rate Limiting

Membrane includes token bucket rate limiting to prevent abuse.

### Configuration

```yaml
rate_limit_per_second: 100  # Max 100 requests per second per client
```

Set to `0` to disable rate limiting:

```yaml
rate_limit_per_second: 0
```

### Rate Limit Exceeded

When rate limits are exceeded, requests fail with `RESOURCE_EXHAUSTED` status:

```
grpc error: code=8 (RESOURCE_EXHAUSTED) message="rate limit exceeded"
```

## Trust-Aware Retrieval

Membrane gates retrieval by sensitivity level, providing graduated access control.

### Sensitivity Levels

| Level | Description | Use Case |
|-------|-------------|----------|
| `public` | Publicly shareable | Public documentation, general knowledge |
| `low` | Internal use | Non-sensitive preferences, metadata |
| `medium` | Restricted | Project-specific data, internal procedures |
| `high` | Confidential | API keys (metadata only), credentials |
| `hyper` | Highly confidential | Secrets, PII, financial data |

### Access Rules

1. Records **at or below** the trust level are returned with full payload
2. Records **one level above** are returned with metadata only (redacted payload)
3. Records **two or more levels above** are not returned

### Example

```ts
import { Sensitivity } from "@gustycube/membrane";

// Caller with MEDIUM trust
const records = await client.retrieve("database credentials", {
  trust: {
    max_sensitivity: Sensitivity.MEDIUM,
    authenticated: true,
    actor_id: "db-agent",
    scopes: ["prod"],
  },
});

// Results:
// - PUBLIC, LOW, MEDIUM records: full payload
// - HIGH records: metadata only (id, type, salience, tags)
// - HYPER records: not returned
```

### Ingestion Sensitivity

Set sensitivity when ingesting records:

<CodeGroup>
```ts TypeScript
await client.ingestEvent("api_key_rotation", "task-1", {
  summary: "Rotated production API key",
  sensitivity: Sensitivity.HIGH,  // Restrict access
  tags: ["security", "credentials"],
});
```

```python Python
client.ingest_event(
    event_kind="api_key_rotation",
    ref="task-1",
    summary="Rotated production API key",
    sensitivity=Sensitivity.HIGH,
    tags=["security", "credentials"],
)
```
</CodeGroup>

## Input Validation

Membrane validates all inputs to prevent injection attacks and resource exhaustion:

- **Payload size limits:** Max 1MB per record
- **String length checks:** Max 10,000 characters for summaries
- **Tag count limits:** Max 50 tags per record
- **NaN/Inf rejection:** Confidence and salience values must be finite

### Validation Errors

Invalid inputs return `INVALID_ARGUMENT` status:

```
grpc error: code=3 (INVALID_ARGUMENT) message="summary exceeds maximum length"
```

## Audit Trails

All revision operations (supersede, fork, retract, merge, contest) and reinforcement actions (reinforce, penalize) create audit log entries.

### Audit Entry Fields

- `id`: Unique audit entry ID
- `record_id`: Target record ID
- `operation`: Operation type (e.g., `supersede`, `reinforce`)
- `actor`: Actor identifier
- `rationale`: Human-readable reason
- `timestamp`: RFC 3339 timestamp

### Querying Audit Logs

Audit logs are stored in the database and can be queried via SQL (future API endpoint planned).

## Best Practices

### 1. Use Environment Variables for Secrets

<Warning>
  Never commit encryption keys or API keys to version control.
</Warning>

```bash
# /etc/membrane/environment
MEMBRANE_ENCRYPTION_KEY=your-encryption-key
MEMBRANE_API_KEY=your-api-key
```

### 2. Enable TLS in Production

Always use TLS for production deployments:

```yaml
tls_cert_file: "/etc/letsencrypt/live/membrane.example.com/fullchain.pem"
tls_key_file: "/etc/letsencrypt/live/membrane.example.com/privkey.pem"
```

### 3. Set Appropriate Sensitivity Levels

Classify records by sensitivity:

```ts
// Low sensitivity: preferences
await client.ingestObservation("user", "prefers_language", "typescript", {
  sensitivity: Sensitivity.LOW,
});

// High sensitivity: credentials
await client.ingestEvent("credential_created", "api-key-1", {
  summary: "Created production API key",
  sensitivity: Sensitivity.HIGH,
});
```

### 4. Use Scopes for Isolation

Scope records to projects or customers:

```ts
await client.ingestEvent("deployment", "task-1", {
  summary: "Deployed to production",
  scope: "project-acme",  // Isolate by project
});

const records = await client.retrieve("deployment history", {
  trust: {
    max_sensitivity: Sensitivity.MEDIUM,
    authenticated: true,
    scopes: ["project-acme"],  // Only retrieve project-acme records
  },
});
```

### 5. Rotate Credentials Regularly

- **Encryption keys:** Rotate annually or after suspected compromise
- **API keys:** Rotate quarterly or after team member departure
- **TLS certificates:** Use automated renewal (e.g., certbot with Let's Encrypt)

### 6. Monitor Failed Authentication Attempts

Track `UNAUTHENTICATED` errors in logs:

```bash
grep UNAUTHENTICATED /var/log/membraned.log
```

### 7. Implement Network Segmentation

Run Membrane on a private network, accessible only to authorized agents:

```yaml
listen_addr: "10.0.1.50:9090"  # Bind to private IP
```

### 8. Backup Encrypted Databases Securely

Store backups with the same encryption strength:

```bash
# Backup (file is already encrypted)
cp /var/lib/membrane/membrane.db /backup/membrane-$(date +%Y%m%d).db

# Store encryption key separately in a secrets manager
```

## Threat Model

### In Scope

- **Unauthorized access to database file** -- Mitigated by encryption at rest
- **Man-in-the-middle attacks** -- Mitigated by TLS transport
- **Unauthorized API access** -- Mitigated by API key authentication
- **Denial of service** -- Mitigated by rate limiting
- **Data leakage via retrieval** -- Mitigated by trust-aware access control

### Out of Scope

- **Memory inspection of running process** -- Use OS-level protections (e.g., memory encryption, SELinux)
- **Physical access to server** -- Use datacenter security controls
- **Compromised client credentials** -- Rotate keys, implement key revocation
