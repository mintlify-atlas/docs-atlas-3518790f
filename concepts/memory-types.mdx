## Overview

Membrane organizes memory into five distinct types, each with its own schema, lifecycle rules, and consolidation behavior. This typed approach prevents the "everything is text" problem common in RAG systems.

```go
type MemoryType string

const (
    MemoryTypeEpisodic   MemoryType = "episodic"
    MemoryTypeWorking    MemoryType = "working"
    MemoryTypeSemantic   MemoryType = "semantic"
    MemoryTypeCompetence MemoryType = "competence"
    MemoryTypePlanGraph  MemoryType = "plan_graph"
)
```

## Memory Type Summary

| Type | Purpose | Lifecycle | Revisable |
|------|---------|-----------|----------|
| **Episodic** | Raw experience capture | Short-lived (days) | No (immutable) |
| **Working** | Current task state | Session-scoped | Yes (frequent updates) |
| **Semantic** | Stable facts and preferences | Long-lived (months) | Yes (evidence-based) |
| **Competence** | Learned procedures | Long-lived (months) | Yes (versioned) |
| **Plan Graph** | Reusable solution structures | Long-lived (months) | Yes (versioned) |

## 1. Episodic Memory

**Purpose:** Capture raw experience as immutable evidence for later learning.

### Schema

From `pkg/schema/payloads.go`:

```go
type EpisodicPayload struct {
    Kind string `json:"kind"` // "episodic"
    
    // Time-ordered sequence of events
    Timeline []TimelineEvent `json:"timeline"`
    
    // Tool calls and data flow during the episode
    ToolGraph []ToolNode `json:"tool_graph,omitempty"`
    
    // Environment snapshot (OS, versions, context)
    Environment *EnvironmentSnapshot `json:"environment,omitempty"`
    
    // Result of the episode (success, failure, partial)
    Outcome OutcomeStatus `json:"outcome,omitempty"`
    
    // External artifacts (logs, screenshots, files)
    Artifacts []string `json:"artifacts,omitempty"`
}

type TimelineEvent struct {
    T         time.Time `json:"t"`          // Timestamp
    EventKind string    `json:"event_kind"` // e.g., "tool_call", "error"
    Ref       string    `json:"ref"`        // External reference
    Summary   string    `json:"summary,omitempty"`
}
```

### Example: Troubleshooting Session

```json
{
  "id": "ep-001",
  "type": "episodic",
  "sensitivity": "low",
  "confidence": 1.0,
  "salience": 0.8,
  "payload": {
    "kind": "episodic",
    "timeline": [
      {
        "t": "2026-03-13T10:00:00Z",
        "event_kind": "tool_call",
        "ref": "build#42",
        "summary": "Executed go build, failed with linker error"
      },
      {
        "t": "2026-03-13T10:05:00Z",
        "event_kind": "tool_call",
        "ref": "clean#43",
        "summary": "Cleared build cache"
      },
      {
        "t": "2026-03-13T10:06:00Z",
        "event_kind": "tool_call",
        "ref": "build#44",
        "summary": "Rebuilt with clean flags, success"
      }
    ],
    "outcome": "success",
    "environment": {
      "os": "darwin",
      "os_version": "14.2",
      "tool_versions": {"go": "1.24.0"}
    }
  }
}
```

### Lifecycle Rules

<Note>
  **Episodic records are immutable** — they cannot be edited after creation. Corrections happen through semantic memory or competence extraction.
</Note>

- **Decay:** Aggressive (half-life: 1-7 days)
- **Pruning:** Auto-prune after salience drops to zero
- **Consolidation:** Extracted into semantic/competence/plan records after successful patterns emerge

## 2. Working Memory

**Purpose:** Track the current state of ongoing tasks for resumption across sessions.

### Schema

From `pkg/schema/payloads.go`:

```go
type WorkingPayload struct {
    Kind string `json:"kind"` // "working"
    
    ThreadID string    `json:"thread_id"` // Session identifier
    State    TaskState `json:"state"`     // planning | executing | blocked | waiting | done
    
    // Active constraints for the task
    ActiveConstraints []Constraint `json:"active_constraints,omitempty"`
    
    // Next planned actions
    NextActions []string `json:"next_actions,omitempty"`
    
    // Unresolved questions
    OpenQuestions []string `json:"open_questions,omitempty"`
    
    // Summary of current context
    ContextSummary string `json:"context_summary,omitempty"`
}

type TaskState string
const (
    TaskStatePlanning  TaskState = "planning"
    TaskStateExecuting TaskState = "executing"
    TaskStateBlocked   TaskState = "blocked"
    TaskStateWaiting   TaskState = "waiting"
    TaskStateDone      TaskState = "done"
)
```

### Example: Project Setup in Progress

```json
{
  "id": "wm-001",
  "type": "working",
  "sensitivity": "low",
  "payload": {
    "kind": "working",
    "thread_id": "session-001",
    "state": "executing",
    "next_actions": [
      "run tests",
      "deploy to staging",
      "update documentation"
    ],
    "context_summary": "Backend initialized with Go, frontend pending framework selection"
  }
}
```

### Lifecycle Rules

- **Decay:** Minimal (half-life: 1-7 days)
- **Updates:** Frequent (updated on every task state change)
- **Deletion:** Auto-prune when task state = "done" and salience decays

## 3. Semantic Memory

**Purpose:** Store stable facts and preferences as revisable subject-predicate-object triples.

### Schema

From `pkg/schema/payloads.go`:

```go
type SemanticPayload struct {
    Kind string `json:"kind"` // "semantic"
    
    Subject   string `json:"subject"`   // Entity the fact is about
    Predicate string `json:"predicate"` // Relationship or property
    Object    any    `json:"object"`    // Value or related entity
    
    // When this fact is valid (global, conditional, timeboxed)
    Validity Validity `json:"validity"`
    
    // Evidence supporting this fact
    Evidence []ProvenanceRef `json:"evidence,omitempty"`
    
    // How to handle revisions (replace, fork, contest)
    RevisionPolicy string `json:"revision_policy,omitempty"`
    
    // Revision tracking
    Revision *RevisionState `json:"revision,omitempty"`
}

type Validity struct {
    Mode       ValidityMode   `json:"mode"` // global | conditional | timeboxed
    Conditions map[string]any `json:"conditions,omitempty"`
    Start      *time.Time     `json:"start,omitempty"`
    End        *time.Time     `json:"end,omitempty"`
}

type RevisionState struct {
    Supersedes   string         `json:"supersedes,omitempty"`
    SupersededBy string         `json:"superseded_by,omitempty"`
    Status       RevisionStatus `json:"status,omitempty"` // active | contested | retracted
}
```

### Example: User Preference with Conditional Validity

```json
{
  "id": "sem-001",
  "type": "semantic",
  "sensitivity": "low",
  "confidence": 0.9,
  "salience": 0.8,
  "payload": {
    "kind": "semantic",
    "subject": "user",
    "predicate": "prefers_backend_language",
    "object": "go",
    "validity": {
      "mode": "conditional",
      "conditions": {"project_kind": "backend_service"}
    },
    "revision": {
      "status": "active"
    }
  }
}
```

<Accordion title="Multiple Conditional Facts">
  Semantic memory supports multiple facts for the same subject-predicate with different conditions:
  
  ```json
  [
    {
      "subject": "user",
      "predicate": "prefers_backend_language",
      "object": "go",
      "validity": {"mode": "conditional", "conditions": {"project_kind": "backend_service"}}
    },
    {
      "subject": "user",
      "predicate": "prefers_backend_language",
      "object": "python",
      "validity": {"mode": "conditional", "conditions": {"project_kind": "data_science"}}
    }
  ]
  ```
</Accordion>

### Lifecycle Rules

- **Decay:** Slow (half-life: 30-90 days)
- **Revisability:** Yes (via supersede, fork, contest, retract)
- **Consolidation:** Extracted from repeated episodic observations

---

## 4. Competence Memory

**Purpose:** Encode procedural knowledge — how to achieve goals reliably under specific conditions.

### Schema

From `pkg/schema/payloads.go`:

```go
type CompetencePayload struct {
    Kind string `json:"kind"` // "competence"
    
    SkillName string `json:"skill_name"`
    
    // When this competence applies
    Triggers []Trigger `json:"triggers"`
    
    // Ordered or conditional steps
    Recipe []RecipeStep `json:"recipe"`
    
    RequiredTools []string `json:"required_tools,omitempty"`
    FailureModes  []string `json:"failure_modes,omitempty"`
    Fallbacks     []string `json:"fallbacks,omitempty"`
    
    // Success/failure statistics
    Performance *PerformanceStats `json:"performance,omitempty"`
    
    Version string `json:"version,omitempty"`
}

type Trigger struct {
    Signal     string         `json:"signal"` // e.g., error signature
    Conditions map[string]any `json:"conditions,omitempty"`
}

type RecipeStep struct {
    Step       string         `json:"step"`
    Tool       string         `json:"tool,omitempty"`
    ArgsSchema map[string]any `json:"args_schema,omitempty"`
    Validation string         `json:"validation,omitempty"`
}
```

### Example: Fix Linker Cache Error

From the README troubleshooting example:

```json
{
  "id": "comp-001",
  "type": "competence",
  "sensitivity": "medium",
  "confidence": 0.7,
  "salience": 0.6,
  "payload": {
    "kind": "competence",
    "skill_name": "fix_linker_cache_error",
    "triggers": [
      {
        "signal": "ld: cache mismatch",
        "conditions": {"os": "macos"}
      }
    ],
    "recipe": [
      {
        "step": "Clear build cache",
        "tool": "shell",
        "validation": "cache directory removed"
      },
      {
        "step": "Rebuild with clean flags",
        "tool": "build",
        "validation": "build succeeds"
      }
    ],
    "failure_modes": [
      "If cache directory is locked, retry with elevated permissions"
    ],
    "fallbacks": ["Reinstall toolchain"],
    "performance": {
      "success_count": 5,
      "failure_count": 1,
      "success_rate": 0.83
    },
    "version": "1.0"
  }
}
```

### Lifecycle Rules

- **Decay:** Moderate (half-life: 30-60 days)
- **Versioning:** Multiple versions can coexist
- **Selection:** Based on trigger match, success rate, and recency
- **Consolidation:** Extracted from repeated successful episodic patterns

## 5. Plan Graph Memory

**Purpose:** Store reusable solution structures as directed graphs of actions.

### Schema

From `pkg/schema/payloads.go`:

```go
type PlanGraphPayload struct {
    Kind string `json:"kind"` // "plan_graph"
    
    PlanID  string `json:"plan_id"`
    Version string `json:"version"`
    
    Intent string `json:"intent,omitempty"` // e.g., "setup_project"
    
    Constraints   map[string]any `json:"constraints,omitempty"`
    InputsSchema  map[string]any `json:"inputs_schema,omitempty"`
    OutputsSchema map[string]any `json:"outputs_schema,omitempty"`
    
    // Graph structure
    Nodes []PlanNode `json:"nodes"`
    Edges []PlanEdge `json:"edges"`
    
    Metrics *PlanMetrics `json:"metrics,omitempty"`
}

type PlanNode struct {
    ID     string         `json:"id"`
    Op     string         `json:"op"` // Action or tool identifier
    Params map[string]any `json:"params,omitempty"`
    Guards map[string]any `json:"guards,omitempty"` // Conditional execution
}

type PlanEdge struct {
    From string   `json:"from"`
    To   string   `json:"to"`
    Kind EdgeKind `json:"kind"` // "data" | "control"
}
```

### Example: Project Setup Workflow

```json
{
  "id": "plan-001",
  "type": "plan_graph",
  "sensitivity": "low",
  "payload": {
    "kind": "plan_graph",
    "plan_id": "setup_go_project",
    "version": "1.0",
    "intent": "setup_project",
    "nodes": [
      {"id": "n1", "op": "init_git"},
      {"id": "n2", "op": "create_go_mod", "params": {"module": "example.com/project"}},
      {"id": "n3", "op": "create_main"},
      {"id": "n4", "op": "setup_ci"}
    ],
    "edges": [
      {"from": "n1", "to": "n2", "kind": "control"},
      {"from": "n2", "to": "n3", "kind": "control"},
      {"from": "n2", "to": "n4", "kind": "control"}
    ],
    "metrics": {
      "execution_count": 12,
      "avg_latency_ms": 8500,
      "failure_rate": 0.08
    }
  }
}
```

### Lifecycle Rules

- **Decay:** Slow (half-life: 60-90 days)
- **Versioning:** Multiple versions coexist
- **Selection:** Based on intent match, constraints, and execution success
- **Consolidation:** Extracted from tool call sequences in episodic memory

## Memory Record Base Structure

All memory types share a common base structure:

From `pkg/schema/memory.go`:

```go
type MemoryRecord struct {
    ID          string          `json:"id"`
    Type        MemoryType      `json:"type"`
    Sensitivity Sensitivity     `json:"sensitivity"`
    Confidence  float64         `json:"confidence"`  // [0, 1]
    Salience    float64         `json:"salience"`    // [0, +inf)
    Scope       string          `json:"scope,omitempty"`
    Tags        []string        `json:"tags,omitempty"`
    CreatedAt   time.Time       `json:"created_at"`
    UpdatedAt   time.Time       `json:"updated_at"`
    Lifecycle   Lifecycle       `json:"lifecycle"`
    Provenance  Provenance      `json:"provenance"`
    Relations   []Relation      `json:"relations,omitempty"`
    Payload     Payload         `json:"payload"`
    AuditLog    []AuditEntry    `json:"audit_log"`
}
```

## Choosing the Right Memory Type

<Tabs>
  <Tab title="Use Episodic When">
    - Capturing raw events, tool calls, observations
    - Recording errors or debugging sessions
    - Preserving immutable evidence for future learning
    - Short-term (days) retention is acceptable
  </Tab>
  
  <Tab title="Use Working When">
    - Tracking current task state for resumption
    - Managing multi-step workflows across sessions
    - Storing temporary context that expires when task completes
  </Tab>
  
  <Tab title="Use Semantic When">
    - Storing user preferences
    - Recording environment facts
    - Capturing relationships between entities
    - Long-term (months) retention is needed
    - Facts may need revision or contestation
  </Tab>
  
  <Tab title="Use Competence When">
    - Learning troubleshooting procedures
    - Encoding multi-step workflows with success tracking
    - Storing "how-to" knowledge with conditions
    - Multiple versions may coexist
  </Tab>
  
  <Tab title="Use Plan Graph When">
    - Capturing complex multi-action workflows
    - Reusing solution templates across similar tasks
    - Tracking dependencies between actions
    - Measuring plan execution success over time
  </Tab>
</Tabs>
